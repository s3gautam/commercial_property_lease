# Database

PostgreSQL via SQLAlchemy 2.x (async) and Alembic migrations. Schema is
never created automatically — every change is a migration under
`services/backend/alembic/versions`.

## Conventions

- Every mutable table extends `TimestampedModel`
  (`services/backend/app/models/base.py`), providing:
  - `id` (UUID primary key)
  - `created_at`, `updated_at`
  - `created_by`, `updated_by`
- Append-only tables (currently just `audit_logs`) extend `ImmutableModel`
  instead: `id`, `created_at`, `created_by` — no `updated_at`/`updated_by`
  since rows are never modified after insert.
- Indexes are added for every foreign key and frequently filtered column.

## Tables (Tenant MVP)

| Table                   | Purpose                              | Model                                         |
| ------------------------ | -------------------------------------- | ------------------------------------------------ |
| `users`                   | Account records across roles          | `app/models/user.py::User`                       |
| `tenant_profiles`         | Tenant-specific profile data          | `app/models/user.py::TenantProfile`              |
| `properties`              | Commercial property listings          | `app/models/property.py::Property`               |
| `property_images`         | Property media                        | `app/models/property.py::PropertyImage`          |
| `property_documents`      | Supporting documents for a property    | `app/models/property.py::PropertyDocument`       |
| `verification_reports`    | AI-generated property verification    | `app/models/property.py::VerificationReport`     |
| `chat_threads`            | Tenant/landlord conversations         | `app/models/chat.py::ChatThread`                 |
| `messages`                | Chat messages                         | `app/models/chat.py::Message`                    |
| `leases`                  | Lease agreements                      | `app/models/lease.py::Lease`                     |
| `lease_versions`          | Lease revision history                | `app/models/lease.py::LeaseVersion`              |
| `kyc_verifications`       | Tenant KYC records                    | `app/models/kyc.py::KYCVerification`             |
| `notifications`           | User notifications                    | `app/models/notification.py::Notification`       |
| `audit_logs`              | System-wide audit trail (append-only) | `app/models/audit_log.py::AuditLog`              |

All 13 tables are created by the initial migration
(`0001_initial_schema`). Its `upgrade()`/`downgrade()` were verified
end-to-end against a real local PostgreSQL 16 instance.

A second migration (`0002_lease_version_document_text`, Phase 5) added
`lease_versions.document_text` (nullable `TEXT`) and made
`lease_versions.document_url` nullable. The AI-drafted lease text
(`LeaseDraftingAgent`) has nowhere to be uploaded yet — there's no S3
client wired up despite MinIO being in `docker-compose.yml` — so it's
stored directly in `document_text` for now. `document_url` will hold a
real object-storage URL once that pipeline exists.

Notable constraints:

- `users.email`/`users.phone` are both nullable, but a `CHECK` constraint
  (`ck_users_email_or_phone`) requires at least one to be set — supporting
  Google OAuth (email only), email OTP, or phone OTP signup.
- `tenant_profiles.user_id` is unique (one profile per user).
- Foreign keys use `ON DELETE CASCADE` for owned child rows (e.g.
  `property_images` -> `properties`) and `ON DELETE SET NULL` where the
  parent is optional context (e.g. `properties.landlord_id` -> `users`,
  since the Landlord Portal doesn't exist yet).

## Migrations

Business tables so far were hand-authored (no live dev database was
available when they were written) rather than autogenerated, and verified
against a real Postgres instance before being committed. Going forward,
generate migrations the standard way:

```bash
cd services/backend
alembic revision --autogenerate -m "add properties search index"
alembic upgrade head
```

Always review autogenerated migrations before committing — Alembic can
miss things like `CHECK` constraints and enum value changes.
